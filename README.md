# Z-Chat — Private Ephemeral Messenger

Zero-budget, end-to-end encrypted messenger with a cinematic dissolve effect.
Cyber-Phantom aesthetic, glassmorphism UI, `tweetnacl` box encryption, and a
zero-knowledge WebSocket relay. Works as a real multi-user app: register a
public key, discover other users, and exchange messages that only the holder
of the matching secret key can decrypt.

## Quick start

```bash
npm run install:all
npm run dev            # Vite :5173 + backend relay :3001
```

- Frontend: http://localhost:5173
- Backend relay: ws://localhost:3001
- Health: http://localhost:3001/health

## Architecture

- **Frontend** (`frontend/`) — React 18 + Vite + Tailwind. Entry: `main.jsx` → `App.jsx` (Router, ErrorBoundary, Theme/Auth/GhostMode providers).
- **Backend** (`backend/`) — Express + `ws` WebSocket relay. Exposes:
  - `POST /api/auth/token` — mints a session token for the relay.
  - `POST /api/users/register` — publishes a user's public key to the shared directory.
  - `GET /api/users?exclude=<id>` — lists users for the contact picker.
  - `GET /api/users/:id` — looks up a single user.
  - `ws://…` — the zero-knowledge relay (only ciphertext passes through).
- **Persistence** — pluggable store:
  - **File store** (default, zero-config): messages + users persisted to `backend/data/` as JSON. Great for local dev and a working single-instance deploy.
  - **Supabase/Postgres** (production-grade): set `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` and the backend auto-switches to durable, queryable storage. Schema in `database/schema.sql`.
- **Encryption** — `tweetnacl` `box` with a unique nonce per message. Keys are generated locally and never leave the device; the backend only ever sees ciphertext.
- Demo contacts are seeded on first run so a fresh deployment isn't empty.

## Key features

- Real E2EE (per-message `tweetnacl` box encryption)
- Shared user directory + contact discovery
- Optimistic UI (messages render before server ACK)
- Ephemeral messages with self-destruct timer + Canvas dissolve particle effect
- Message status tracking (sending → sent → delivered → read)
- Ghost Mode (appear offline, suppress typing indicators)
- Offline message store with deliver-on-reconnect
- Typing indicator, presence, encryption fingerprint verification
- Glassmorphism + neon Cyber-Phantom design system

## Testing

```bash
npm test              # frontend unit tests (Vitest)
cd backend && npm test   # backend unit tests (Vitest)
npm run test:e2e      # Playwright E2E (real browser, spins up both servers)
```

CI (`.github/workflows/ci.yml`) runs backend + frontend unit tests, the build,
and the E2E suite on every push.

## Deploy

### 1. Backend → Render

`render.yaml` is a ready-to-use blueprint (rootDir `backend`, `node src/server.js`).
Import the repo in Render, or push and connect. Set:

- `JWT_SECRET` — auto-generated by Render.
- `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` — **optional**; without them the
  backend uses the file store. With them it uses durable Postgres.

Note the `repo:` line in `render.yaml` is a placeholder — Render fills this in
when you create the service from the dashboard.

### 2. Frontend → Vercel

`frontend/vercel.json` handles the SPA build. Import the repo in Vercel
(framework = Vite). Set these environment variables so the client can reach the
backend in production:

- `VITE_API_URL` — e.g. `https://z-chat-backend.onrender.com`
- `VITE_WS_URL` — e.g. `wss://z-chat-backend.onrender.com`
- `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` — optional (only if you wire the
  frontend Supabase client).

### 3. Database (optional)

If using Supabase: `supabase db push` with `database/schema.sql`. The
`message_id` column is used to track delivery/read receipts.

## Notes / roadmap

- Encryption today uses a static identity keypair with a unique nonce per message
  (secure E2EE). A `doubleRatchet` utility module is included for forward-secrecy
  experiments but is not yet wired into the send path.
- Not yet implemented: group chats, media/file messages, message editing, push
  notifications.
